Yazılım geliştirme dünyasında kod yazmak, sadece çalışan bir ürün ortaya koymaktan ibaret değildir. Yazdığımız kodun sürdürülebilir, test edilebilir ve gelecekte kolayca değiştirilebilir olması gerekir. İşte tam bu noktada Clean Architecture devreye giriyor.

Clean Architecture Nedir?

Clean Architecture, Robert C. Martin (Uncle Bob) tarafından tanımlanan bir yazılım mimari yaklaşımıdır. Temel amacı, yazılımın iş mantığını dış dünya ile olan bağımlılıklardan ayırmaktır. Bu sayede:

  • Kod daha test edilebilir hale gelir
  • Framework'lerden bağımsız çalışır
  • UI değişikliklerinden etkilenmez
  • Database değişikliklerine dayanıklıdır
  • Dış servislerden bağımsızdır

Önemli Not

Clean Architecture, projeye başlarken uygulanması en kolay olan mimaridir. Projeyi yarıda temiz koda geçirmek çok daha zordur.

Temel Prensipler

1. Dependency Rule (Bağımlılık Kuralı)

En önemli kural: Kaynak kod bağımlılıkları sadece içeriye doğru işaret edebilir. İç katmanlar dış katmanlar hakkında hiçbir şey bilmez. 

// ❌ Yanlış: Domain katmanı framework'e bağımlı
class User {
    use HasFactory; // Laravel'e bağımlılık
}

// ✅ Doğru: Domain katmanı bağımsız
class User {
    private string $name;
    private string $email;
    
    public function __construct(string $name, string $email) {
        $this->name = $name;
        $this->email = $email;
    }
}

2. Entities (Varlıklar)

Entities, işletme kurallarını içeren katmandır. Tüm uygulamada paylaşılan iş kuralları burada yaşar. 

class Order {
    private array $items;
    private string $status;
    
    public function calculateTotal(): float {
        return array_reduce($this->items, 
            fn($total, $item) => $total + $item->getPrice(), 
            0
        );
    }
    
    public function canBeCancelled(): bool {
        return $this->status === 'pending' || 
               $this->status === 'confirmed';
    }
}

3. Use Cases (Kullanım Senaryoları)

Use Cases, uygulamaya özgü iş kurallarını içerir. Sistemin tüm kullanım senaryolarını orkestra eder. 

class CreateOrderUseCase {
    public function __construct(
        private OrderRepository $repository,
        private PaymentGateway $paymentGateway,
        private EmailService $emailService
    ) {}
    
    public function execute(CreateOrderRequest $request): Order {
        $order = new Order($request->items, $request->userId);
        
        $this->paymentGateway->charge($order->getTotal());
        $this->repository->save($order);
        $this->emailService->sendOrderConfirmation($order);
        
        return $order;
    }
}

Neden Öğrenmeliyiz?

1. Uzun Vadeli Sürdürülebilirlik

Bir proje başlangıçta basit görünebilir, ancak zamanla karmaşıklaşır. Clean Architecture ile yazdığınız kod, 5 yıl sonra bile rahatça anlaşılabilir ve değiştirilebilir.

2. Test Edilebilirlik

İş mantığınız framework'lerden ve external servislerden bağımsız olduğu için, unit testler yazmak çok daha kolaydır. 

class OrderTest extends TestCase {
    public function test_order_total_calculation() {
        $order = new Order([
            new OrderItem('Product 1', 100),
            new OrderItem('Product 2', 200)
        ]);
        
        $this->assertEquals(300, $order->calculateTotal());
    }
    
    // Framework'e ihtiyaç yok!
}

3. Framework Bağımsızlığı

Laravel bugün, başka bir framework yarın. Clean Architecture ile iş mantığınızı framework değişikliklerinden korursunuz.

"Framework'ler araçtır, amaç değildir. İş mantığınızı araçlara bağımlı kılmayın." - Robert C. Martin

Pratik Uygulamaya Nasıl Başlamalı?

Önerilen Yol Haritası

  1. SOLID Prensiplerini öğrenin: Clean Architecture'ın temeli SOLID'dir
  2. Küçük projelerle başlayın: Todo app gibi basit projelerle pratik yapın
  3. Dependency Injection kullanın: Bağımlılıkları yönetmeyi öğrenin
  4. Repository Pattern öğrenin: Data katmanını soyutlamayı öğrenin
  5. Use Case'leri tanımlayın: İş mantığınızı organize edin

Sonuç

Clean Architecture öğrenmek başlangıçta zor görünebilir, ancak uzun vadede size muazzam avantajlar sağlar. Özellikle profesyonel projelerde çalışırken, bu mimari yaklaşımı bilmek sizi diğer geliştiricilerden ayıracaktır.

Unutmayın: Clean Architecture bir dogma değil, bir rehberdir. Projenizin ihtiyaçlarına göre uyarlayabilirsiniz. Önemli olan prensipleri anlamak ve doğru yerde uygulamaktır.
